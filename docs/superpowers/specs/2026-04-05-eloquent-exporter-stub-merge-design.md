# Design: Eloquent Exporter — Artisan Stub Merge

**Date:** 2026-04-05
**Status:** Approved

## Problem

`EloquentExporter` generates PHP Eloquent model files with hardcoded assumptions about Laravel version (namespace `App\Models`, no `HasFactory`, etc.). When the generated files are dropped into a Laravel project, they may be incompatible with the project's version-specific boilerplate conventions. Maintaining version-specific templates in Archivolt is ongoing maintenance overhead.

## Goal

Make Eloquent export via `--laravel <path>` immune to Laravel version differences by delegating boilerplate generation to `php artisan make:model`, and having Archivolt only inject schema-derived content (table name, fillable columns, relationships).

## Design

### Core Idea

`LaravelArtisanWriter` becomes a two-phase writer:

1. **Bootstrap**: run `artisan make:model <Name>` to create a version-correct stub in the right location
2. **Read**: parse the generated stub to extract `namespace` and existing traits (e.g. `HasFactory`)
3. **Merge**: pass a `StubContext` to `EloquentExporter.renderModel()`, which uses the artisan-provided values instead of hardcoded defaults
4. **Write**: overwrite the stub with the fully merged content

### New Interface: `StubContext`

```typescript
interface StubContext {
  readonly namespace: string        // e.g. "App\\Models"
  readonly existingTraits: string[] // e.g. ["HasFactory"]
  readonly filePath: string         // absolute path artisan wrote to
}
```

### `LaravelArtisanWriter` Changes

- After `artisan make:model`, read the generated file
- Extract `namespace` via `/^namespace\s+([\w\\]+);/m`
- Extract traits via `/^\s+use\s+([\w\\]+);/gm` (filters to short class names, excludes FQCN imports)
- Derive `filePath` from the regex match or from the known Laravel convention (`app/Models/<Name>.php`)
- Pass `StubContext` to `EloquentExporter.renderModel()`

### `EloquentExporter` Changes

`renderModel()` accepts an optional `stubContext?: StubContext` parameter:

- **With** `stubContext`: use `stubContext.namespace` for the namespace declaration; prepend `stubContext.existingTraits` (e.g. `HasFactory`) before any Archivolt-added traits (e.g. `SoftDeletes`)
- **Without** `stubContext` (stdout / `--output` modes): current behavior unchanged — defaults to `App\Models` namespace, no `HasFactory`

### Stub Parsing Rules

Only two values are extracted from the artisan stub:

| Value | Regex | Example result |
|-------|-------|----------------|
| namespace | `/^namespace\s+([\w\\]+);/m` | `"App\\Models"` |
| traits in class body | `/^\s+use\s+([\w\\]+);/gm` — short names only | `["HasFactory"]` |

All other content (properties, methods, relationships) is regenerated by `renderModel()`. The artisan stub is treated as disposable after extraction.

## Scope

### In scope
- Refactor `LaravelArtisanWriter` to extract `StubContext` after `artisan make:model`
- Update `EloquentExporter.renderModel()` to accept and use `StubContext`
- Unit tests for stub parsing logic
- Unit tests for `renderModel()` with and without `StubContext`

### Out of scope
- Changes to `IExporter` interface
- Changes to other exporters (Mermaid, DBML, Prisma)
- stdout / `--output` mode behavior for Eloquent
- Any new CLI flags
- Full PHP AST parsing

## Testing Strategy

**`LaravelArtisanWriter` tests:**
- Mock `exec` (already done)
- New: parse stub fixture strings and assert extracted `StubContext` values
- New: assert that `filePath` is correctly derived

**`EloquentExporter` tests:**
- Existing tests cover no-context path (no change needed)
- New: tests with `StubContext` assert that namespace and `HasFactory` trait appear correctly in output

## Risks

- Developer-customized stubs (unusual formatting, `implements` clauses) could cause regex to miss the namespace or traits — mitigation: fall back to `App\Models` and empty traits array rather than throwing
